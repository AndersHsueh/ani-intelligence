# Ani 中远期演进规划

参考 [Scaling Managed Agents: Decoupling the Brain from the Hands](https://www.anthropic.com/engineering/managed-agents)

接口边界（`SessionStore` / `IToolExecutor` / `BaseProvider`）已在 v0.4.0 落地。以下是后续演进方向。

---

## 中期：进程隔离与安全边界

### 工具执行安全隔离

**背景**：工具执行和 LLM 当前共享同一进程空间。接入 MCP 或自定义工具时，prompt injection 和凭据泄露风险上升。`IToolExecutor` interface 已就绪，实现可随时替换。

**待办**：
- [ ] MCP 工具通过独立子进程运行，OAuth token 不进入主进程
- [ ] 建立凭据隔离准入清单：明确哪些工具需要敏感凭据
- [ ] 实现 `SandboxedToolExecutor`，将工具调用序列化后发送到子进程执行

### 进程故障恢复（不要养宠物）

**背景**：Provider 故障已降级为错误事件（不崩 TUI），但推理循环本身没有重启能力。Session 已持久化（v0.4.0），断点恢复的基础设施已就绪。

**待办**：
- [ ] 封装 `Harness` interface，使推理循环可从任意 session 断点独立重启
- [ ] 评估进程间通信方案：单进程内事件机制 vs. 未来分进程的 Unix socket / IPC

```ts
interface Harness {
  run(sessionId: string): Promise<void>
  abort(): void
}
```

---

## 远期：多大脑 / 多双手

**背景**：当前单 provider + 单工具注册表，不支持多模型协同或工具执行卸载。前两个中期目标（进程隔离 + Harness 解耦）是先决条件。

**待办**：
- [ ] `ToolExecutor` 支持路由：按工具名/类型将调用分发到不同执行环境
- [ ] `ProviderFactory` 支持按任务动态选择模型（如推理用 Claude，轻量任务用本地 Ollama）
- [ ] 多大脑协调策略（任务分发、结果聚合）作为独立模块，不内嵌到 Harness
