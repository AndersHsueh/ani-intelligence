# Ani 项目改进计划 — 2026-05-12

## 背景

参考 [Scaling Managed Agents: Decoupling the Brain from the Hands](https://www.anthropic.com/engineering/managed-agents)，从 Anthropic 工程实践中提炼出的架构思路，对 Ani 项目的指导意义。

---

## 核心诊断

Ani 当前最大的技术债**不是功能缺失，而是缺乏接口边界**。

`LLMClient`、`ToolExecutor`、`Session` 三者目前互相感知、紧耦合。只要把这三者的接口各自收紧为显式 interface，之后所有演进（持久化、隔离、多模型）都可以在不破坏 TUI 的前提下逐步进行。

> 核心原则：先定义好 Session / Harness / Sandbox 的边界，不必现在就实现全部，但接口要留好。

---

## 演进路径（按依赖顺序）

```
1. Session 持久化          ← 独立，可最先做
2. 工具执行安全边界         ← 独立，可与 1 并行
3. 进程故障恢复（解耦 Harness）  ← 依赖 1（需要断点恢复能力）
4. 多大脑 / 多双手          ← 依赖 1 + 3（需要稳定的 Session 和进程边界）
```

---

## 1. Session 持久化

### 现状
Session 保存在内存中，进程崩溃或重启后对话历史完全丢失。

### 目标
把 Session 从内存提升为外部持久存储的事件日志，通过稳定接口让大脑从任意断点恢复。

```ts
interface SessionStore {
  getSession(id: string): Promise<Session>
  emitEvent(id: string, event: SessionEvent): Promise<void>
  listSessions(): Promise<SessionMeta[]>
}
```

先定义接口，实现可以从最简单的 JSON 文件开始，之后替换为 SQLite 而不影响上层。

### 待办
- [ ] 在 `src/session.ts` 中提取 `SessionStore` interface，与实现分离
- [ ] 实现 `FileSessionStore`（JSON 落盘），作为默认实现
- [ ] 在 `LLMClient` 中注入 `SessionStore`，不再直接持有内存 Map
- [ ] 持久化方案验证：JSON 文件是否足够，还是需要 SQLite？

## 2. 工具执行安全边界

### 现状
工具执行和 LLM 共享同一进程空间，没有隔离。Prompt injection 风险在接入自定义工具或 MCP 时尤为突出。

### 目标
建立安全边界：Claude 生成的不受信任代码与凭据之间物理隔离，工具执行遵循最小权限原则。

```ts
interface ToolExecutor {
  execute(name: string, input: unknown): Promise<ToolResult>
  listTools(): ToolDefinition[]
}
```

Harness（大脑）只通过这个接口调用工具，不知道工具运行在哪个进程里。

### 待办
- [ ] 在 `src/tools/executor.ts` 中提取 `ToolExecutor` interface
- [ ] 现有 builtin 工具实现为 `LocalToolExecutor`（当前行为不变）
- [ ] 明确哪些工具需要敏感凭据，建立凭据隔离的准入清单
- [ ] MCP 工具接入时，通过独立子进程运行，OAuth token 不进入主进程

---

## 3. 进程故障恢复（不要养宠物）

### 现状
Ani 是单进程模型，LLM provider 故障或某个工具 panic 会导致整个 TUI 崩溃。没有进程级别的故障恢复机制。

### 目标
将 Harness（推理循环）和执行环境解耦。崩溃的组件可以被重启替换，而非依赖人工修复。Session 持久化（第1点）是这一目标的前提——没有断点恢复，重启就等于从零开始。

```ts
interface Harness {
  run(sessionId: string): Promise<void>  // 可从任意 session 断点启动
  abort(): void
}
```

### 待办
- [ ] 在 `LLMClient` 或其上层封装 `Harness` interface，使推理循环可独立重启
- [ ] Provider 故障时捕获异常，向 Session 写入错误事件，而非直接 throw 崩溃 TUI
- [ ] 评估进程间通信方案：当前单进程内足够用事件机制；未来分进程可选 Unix socket / IPC

---

## 4. 多大脑 / 多双手的扩展性

### 现状
单一 provider + 单一工具注册表的紧耦合设计，不支持多模型协同或工具执行卸载。

### 目标
支持多 provider 并发调用（多大脑）、以及工具执行路由到独立进程（多双手）。前三点的接口边界是这一步的基础设施。

```ts
interface LLMProvider {
  chatStreamWithTools(messages, tools, config): AsyncIterable<LLMEvent>
}
// ProviderFactory 按任务/配置动态选择实现
```

### 待办
- [ ] 整理 `src/core/providers/` 中现有 provider，统一到 `LLMProvider` interface
- [ ] `ToolExecutor` 支持路由：按工具名/类型将调用分发到不同执行环境
- [ ] 多大脑协调策略（任务分发、结果聚合）：作为独立模块，不内嵌到 Harness

---

## 近期行动（保守路径，不破坏现有功能）

按优先级，可立即开始：

1. **提取三个 interface**（不改实现）：`SessionStore` / `ToolExecutor` / `LLMProvider`
2. **实现 `FileSessionStore`**，替换 `src/session.ts` 中的内存 Map
3. **Provider 异常捕获**：把 LLM 故障从"进程崩溃"降级为"会话错误事件"

这三步都可以在单进程架构内完成，不需要引入子进程或新依赖。
